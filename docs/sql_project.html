SQL Project: Competitive Pokémon Database

  • Show All Code

Team project with Martin Tran for STAT 405, Advanced Data Computing. Full code and data on GitHub.

Overview

Using SQL (run through a local duckdb database in R), we built a relational database from three competitive-Pokémon datasets — pokemon, moves, and pokemon_info — covering every Pokémon up to Generation 8, then queried it to answer questions about competitive tiers, typings, base stats, and move power.

One result we liked: even though Normal has the single highest-powered move in the game, Fire moves have the highest median power — Fire is consistently strong, while Normal’s average is dragged down by many weak moves.

Database setup

Rather than letting duckdb infer the schema automatically, we define each table explicitly — typing every field, setting defaults, and assigning a primary key — then load the CSVs with COPY. This makes the structure of the data deliberate and reproducible.

The full construction is in the dropdown below.

library(tidyverse)
library(duckdb)    # build a duckdb database locally

con_pokemon <- DBI::dbConnect(duckdb::duckdb(), dbdir = "pokemon_db")
USE pokemon_db;
DROP TABLE IF EXISTS pokemon;
CREATE TABLE pokemon (
  name       VARCHAR(255) NOT NULL DEFAULT ' ',  -- competitive name; unique, so primary key
  hp         SMALLINT NOT NULL DEFAULT 0,         -- base stats top out ~255, SMALLINT is plenty
  atk        SMALLINT NOT NULL DEFAULT 0,
  def        SMALLINT NOT NULL DEFAULT 0,
  spa        SMALLINT NOT NULL DEFAULT 0,
  spd        SMALLINT NOT NULL DEFAULT 0,
  spe        SMALLINT NOT NULL DEFAULT 0,
  weight     DOUBLE NOT NULL DEFAULT 0,           -- kg, has decimals -> DOUBLE
  height     DOUBLE NOT NULL DEFAULT 0,           -- m, has decimals -> DOUBLE
  formats    VARCHAR(255),                        -- competitive tier (OU, UU, NFE, LC...)
  generation VARCHAR(255),                        -- game generation code (RB, etc.)
  PRIMARY KEY (name)
);
COPY pokemon FROM 'pokemon_df.csv' (FORMAT CSV, HEADER, NULL 'NA');
DROP TABLE IF EXISTS moves;
CREATE TABLE moves (
  name        VARCHAR(255) NOT NULL DEFAULT ' ',  -- move name; unique, so primary key
  category    VARCHAR(255),                        -- Physical / Special / Non-Damaging
  power       SMALLINT NOT NULL DEFAULT 0,         -- base power; 0 for status moves
  accuracy    SMALLINT NOT NULL DEFAULT 0,         -- 0-100
  priority    SMALLINT NOT NULL DEFAULT 0,         -- can be negative; SMALLINT is signed
  pp          SMALLINT NOT NULL DEFAULT 0,         -- power points
  description VARCHAR(255),                         -- free-text effect
  type        VARCHAR(255),                         -- elemental type
  PRIMARY KEY (name)
);
COPY moves FROM 'moves_df.csv' (FORMAT CSV, HEADER, NULL 'NA');
DROP TABLE IF EXISTS pokemon_info;
CREATE TABLE pokemon_info (
  id              INTEGER NOT NULL DEFAULT 0,        -- unique row id, so primary key
  pokemon         VARCHAR(255) NOT NULL DEFAULT ' ', -- name; links to pokemon.name
  species_id      INTEGER NOT NULL DEFAULT 0,
  height          DOUBLE NOT NULL DEFAULT 0,
  weight          DOUBLE NOT NULL DEFAULT 0,
  base_experience SMALLINT NOT NULL DEFAULT 0,
  type_1          VARCHAR(255),                      -- primary type
  type_2          VARCHAR(255) DEFAULT NULL,         -- single-type pokemon have none -> NULL
  hp              SMALLINT NOT NULL DEFAULT 0,
  attack          SMALLINT NOT NULL DEFAULT 0,
  defense         SMALLINT NOT NULL DEFAULT 0,
  special_attack  SMALLINT NOT NULL DEFAULT 0,
  special_defense SMALLINT NOT NULL DEFAULT 0,
  speed           SMALLINT NOT NULL DEFAULT 0,
  color_1         VARCHAR(255),                      -- hex colour
  color_2         VARCHAR(255) DEFAULT NULL,         -- optional, so nullable
  color_f         VARCHAR(255) DEFAULT NULL,
  egg_group_1     VARCHAR(255),
  egg_group_2     VARCHAR(255) DEFAULT NULL,         -- many have a single egg group -> NULL
  url_icon        VARCHAR(255),
  generation_id   SMALLINT DEFAULT NULL,             -- 1-8; regional forms are NULL
  url_image       VARCHAR(255),
  PRIMARY KEY (id)
);
COPY pokemon_info FROM 'pokemon_info_df.csv' (FORMAT CSV, HEADER, NULL 'NA');

Querying the database

With the database in place, we can answer questions that would be tedious to look up by hand, those being filtering Pokémon by tier, typing, stats, and more. Each query below runs directly against the database.

The heaviest Pokémon in the OU tier

Code 
-- heaviest pokemon in the OU (overused) competitive format (Muizz)
SELECT name, weight, formats
FROM pokemon
WHERE formats = 'OU'
ORDER BY weight DESC
LIMIT 1;

Strongest Pokémon in OU, by total base stats

Code 
-- base stat total of pokemon in OU tier, highest to lowest (Martin)
SELECT pokemon, base_experience, formats, (pokemon_info.hp + pokemon_info.attack + pokemon_info.defense + pokemon_info.special_attack + pokemon_info.special_defense + pokemon_info.speed) AS total
FROM pokemon_info
JOIN pokemon ON pokemon.name = pokemon_info.pokemon
WHERE formats = 'OU'
ORDER BY total DESC
LIMIT 10;

The tallest Pokémon in each generation

Code 
-- tallest pokemon in each generation (Muizz)
-- Uses a correlated subquery: for each row, compare its height to the MAX height
-- within that same generation, keeping only the matches.
SELECT generation_id, pokemon, height
FROM pokemon_info AS p1
WHERE height = (
        SELECT MAX(height)
        FROM pokemon_info AS p2
        WHERE p1.generation_id = p2.generation_id
      )
  AND generation_id IS NOT NULL;   -- regional forms have no listed generation (real NULLs)

Average base-stat total by generation

SELECT generation_id,
       AVG(hp + attack + defense + special_attack + special_defense + speed) AS total

GROUP BY generation_id;

Plot: average stat total per generation 
generation_plot <- DBI::dbGetQuery(con_pokemon,
  "SELECT generation_id,
          AVG(hp + attack + defense + special_attack + special_defense + speed) AS total

ggplot(generation_plot, aes(x = generation_id, y = total)) +

## Querying the database

With the database in place, we can answer questions that would be tedious to look
up by hand, those being filtering Pokémon by tier, typing, stats, and more. Each query below runs directly against the database.

### The heaviest Pokémon in the OU tier

```{sql connection=con_pokemon}
-- heaviest pokemon in the OU (overused) competitive format (Muizz)
SELECT name, weight, formats
FROM pokemon
WHERE formats = 'OU'
ORDER BY weight DESC
LIMIT 1;
```

*The single heaviest Pokémon among those competing in the top "OverUsed" tier.*

### Strongest Pokémon in OU, by total base stats

```{sql connection=con_pokemon}
-- base stat total of pokemon in OU tier, highest to lowest (Martin)
SELECT pokemon, base_experience, formats, (pokemon_info.hp + pokemon_info.attack + pokemon_info.defense + pokemon_info.special_attack + pokemon_info.special_defense + pokemon_info.speed) AS total
FROM pokemon_info
JOIN pokemon ON pokemon.name = pokemon_info.pokemon
WHERE formats = 'OU'
ORDER BY total DESC
LIMIT 10;
```

*Joins the two tables and sums the six base stats on the fly to rank the most
statistically powerful Pokémon in the OU tier.*

### The tallest Pokémon in each generation

